User Manual

Tesla Solar Charger automatically manages your Tesla EV charging to maximize solar self-consumption. It reads real-time data from your inverter via MQTT (Solar Assistant), calculates available surplus solar power, and dynamically adjusts charging current — either through the Tesla Fleet API or directly via an ESPHome BLE bridge (no internet required).

1. System Overview

The app supports two vehicle communication channels. Fleet API uses Tesla's cloud; BLE communicates directly with your Tesla via an ESP32 Bluetooth bridge.

Solar Panels
→
Deya Inverter
→
MQTT / Solar Assistant
→
This App
→
Fleet API Proxy or ESPHome BLE
→
Tesla EV

Data Source

Solar Assistant publishes inverter, battery, and grid data to MQTT every ~5 seconds. The app subscribes to all solar_assistant/# topics and caches values in Redis.

Decision Engine

Every 10 seconds, the algorithm calculates surplus solar power and adjusts Tesla charging current up or down to keep grid import and battery discharge minimal.

Fleet API Channel

Commands are signed by tesla-http-proxy using your Fleet API private key and sent via the Vehicle Command Protocol. Poll interval ~300s recommended due to API rate limits (~200 req/day).

BLE Channel

An ESP32 running ESPHome talks to the Tesla directly over Bluetooth. Commands respond in <1s, no rate limits, no internet dependency. Poll interval can be as low as 10s.

2. Installation from GitHub

Prerequisites

  • Linux server (Ubuntu 22.04+ recommended) or Raspberry Pi 4/5 on your local network
  • Docker and Docker Compose installed (docker compose version should return v2.x)
  • Git installed
  • A running Solar Assistant instance publishing to MQTT
  • A Tesla vehicle with Fleet API access or an ESPHome BLE bridge (ESP32)

Step 4: Start the Application

docker compose up -d

The app will be available at http://<server-ip>:8000. On first boot it creates the SQLite database and seeds default settings and strategies.

Step 5: Verify MQTT Connection

Open the Dashboard. If solar, battery, and grid gauges are showing live values, MQTT is connected. If everything shows 0, check container logs:

docker logs tesla-solar-charger --tail 50

Step 6: Connect Your Tesla

Option A — Fleet API: Go to Settings → Tesla Connection and complete the OAuth flow. See Tesla Fleet API Setup for detailed steps.

Option B — ESPHome BLE: Go to Settings → Communication Channel, select BLE, and enter your ESP32's IP address. See ESPHome BLE Setup.

Step 7: Configure Charging Settings

Go to Settings and review:

  • Charger Phases and Charger Voltage — must match your installation.
  • Min/Max Charge Amps — set based on your charger's capabilities.
  • Min Battery SoC — the algorithm won't charge the EV if home battery is below this.
  • Switch to the appropriate Settings Strategy (Fleet API Default or BLE Aggressive).

Updating to a New Version

git pull
./deploy.sh

The deploy.sh script rebuilds the Docker image, restores certificates and keys, and restarts the containers. Always use it instead of running Docker commands directly.

Verify MQTT Connection

Open the Dashboard. If solar, battery, and grid gauges are showing live values, MQTT is connected. If everything shows 0, check that your MQTT broker is reachable and Solar Assistant is publishing.

2

Connect Your Tesla

Go to Settings. Under Communication Channel, choose Fleet API or BLE. For Fleet API, click Open Tesla Login to complete OAuth. For BLE, enter your ESP32 IP address.

3

Check Charger Phases (1 or 3), Charger Voltage, Min/Max Charge Amps, and Min Battery SoC. These affect all charging calculations.

Plug in your Tesla and click Solar Only on the Dashboard. The algorithm starts automatically when sufficient surplus solar power is available.

7. Charging Modes

Solar Only

The main mode. The algorithm runs every 10 seconds, calculating surplus solar and adjusting charging amps to minimize grid import and battery discharge. Charging starts only when there's enough surplus. Stops when surplus disappears.

Schedule

Charges during a configured time window at a fixed amp rate. Falls back to the algorithm's limits (min/max amps, battery SoC checks). Useful for cheap night-rate electricity or predictable routines.

Manual

Algorithm is suspended. You control charging entirely via the Dashboard buttons (Start, Stop, Set Amps). Pressing Start/Stop on the Dashboard automatically switches to this mode.

Paused

Algorithm is suspended and the car is not commanded. Useful when you want to temporarily halt automatic control without stopping charging manually.

8. How the Algorithm Works

The charging algorithm runs every 10 seconds in Solar Only mode. Here is the decision sequence:

1

Gate Checks

Battery SoC gate: If home battery SoC is below Min Battery SoC, charging is blocked. Resumes after SoC recovers by SoC Hysteresis amount.

Emergency grid gate: If grid import exceeds Emergency Grid Limit, charging stops immediately regardless of other conditions.

Lux gate: If Home Assistant lux is below Lux Stop Threshold, charging is suspended (too dark/overcast to be useful).

2

Surplus Calculation

Available surplus (watts) = PV power − House load − Battery protection buffer

This represents power available to the EV without drawing from the grid or discharging the battery.

3

Penalty System

If grid import exceeds Max Grid Import for longer than Grid Penalty Delay, a penalty reduces the target amps. Clears after grid drops below the limit for Grid Recovery Delay.

If battery discharge exceeds Battery Discharge Threshold for longer than Battery Penalty Delay, a similar penalty applies. Clears after Battery Recovery Delay.

At high battery SoC (above High SoC Threshold), minor discharge up to High SoC Discharge Allow watts is tolerated, suppressing unnecessary ramp-downs.

4

Speculative Start

When inverters have spare solar capacity (PV curtailment), the apparent surplus is low even though more power is available. If battery SoC ≥ Speculative Start SoC and PV production ≥ Speculative Min PV, the algorithm starts charging at minimum amps to probe whether the inverter responds by generating more. If lux is above Lux Aggressive Threshold, the SoC threshold is lowered by 10%.

5

Lux Model (Curtailment Prediction)

If configured, the app learns the historical relationship between lux and PV output over Lux Model Window days. When the inverter appears curtailed, a fraction (Lux Model Curtailment Factor) of the predicted headroom is added to the surplus estimate. A value of 0 disables the model; 0.7 is conservative; 1.0 trusts the model fully.

6

Amp Targeting and Ramping

Target amps = surplus ÷ (phases × voltage). Amps are ramped toward the target in steps no larger than Ramp Up Step or Ramp Down Step. After each change, the algorithm waits Ramp Up Delay or Ramp Down Delay seconds before the next adjustment. This prevents oscillation from momentary cloud cover.

7

Vehicle Commands

Fleet API: Commands are sent to the vehicle via the signed HTTP proxy. The proxy is recreated before each command to prevent IV counter corruption. A minimum 30s gap between commands is enforced. Poll interval is 300s by default (configurable).

BLE: Commands are posted directly to the ESPHome REST API on the ESP32. Response time is typically <1 second. No rate limits. Poll interval can be as low as 10s. Multiple commands can be sent in sequence without waiting.

10. Settings Strategies

A Strategy is a named bundle of all charger settings (ramp delays, thresholds, poll interval, etc.) saved together. Switching strategies applies all values at once — no need to change each setting individually.

Built-in Strategies

Fleet API Default

Conservative settings tuned for the Fleet API rate limits. Poll interval 300s, ramp-up delay 120s, ramp-down 60s. Suitable for most setups when using cloud-based control.

BLE Aggressive

Fast settings for the ESPHome BLE channel. Poll interval 30s, ramp-up delay 30s, ramp-down 20s. Responds quickly to solar changes. Requires BLE channel to be active.

Managing Strategies

On the Settings page, under Settings Strategies:

  • Activate — makes the strategy active; all its values are immediately written to the settings table and used by the algorithm.
  • Duplicate — creates a copy you can rename and modify without affecting the original.
  • Edit — change any setting within the strategy (but it won't take effect until the strategy is activated).
  • Delete — removes the strategy (cannot delete the currently active one).
  • Create New — starts a blank strategy; fill in each value.

Recommended Workflow

When switching communication channels, also switch the matching strategy:

ChannelRecommended Strategy
Fleet APIFleet API Default (or a duplicate with your tweaks)
ESPHome BLEBLE Aggressive (or a duplicate with your tweaks)

11. Scheduling

Schedules let you automatically charge during a specific time window (e.g., cheap night-rate electricity). Configure them on the Settings page under Schedules.

Schedule Parameters

FieldDescription
NameA human-readable label for this schedule
Start TimeHH:MM in 24-hour format when charging begins
End TimeHH:MM when charging stops (can span midnight)
Days of WeekComma-separated 3-letter day codes: mon,tue,wed,thu,fri,sat,sun
Charge AmpsFixed current to use during the scheduled window
Charge LimitOptional: set a charge limit on the vehicle at schedule start
EnabledToggle without deleting the schedule

How It Works

When a schedule is active, the mode switches to Schedule automatically. The algorithm still respects the Emergency Grid Limit and Min Battery SoC gates. When the window ends, mode returns to the previous setting. Schedules do not override Manual mode — if you manually switch to Manual during a schedule, the schedule will not re-activate until the next window.

12. Logger

The Logger tab shows a real-time event log of all workflow events. Events are stored in an in-memory ring buffer (last 500 events) and are cleared on app restart.

Event Categories

CategoryEvents
System Worker started/stopped, mode changes, control loop errors
Algorithm Charging decisions: surplus calculations, amps increased/decreased, insufficient surplus, battery lockout
Tesla Commands sent (set amps, start/stop charging), wake-up attempts, command success/failure, channel status
MQTT Broker connection/disconnection events
Schedule Schedule activation/deactivation
Manual All user-initiated commands: start/stop charging, set amps, wake, lock/unlock, climate, trunk, flash lights, honk, sentry mode, windows, seat heaters, defrost, charge limit, charge port. Logs both the intent and the success/failure result.

Features

  • Category filters: Click a category button to show only events of that type. Click "All" to show everything.
  • Auto-refresh: Enabled by default, polls for new events every 5 seconds. Toggle the checkbox to disable.
  • Clear: Removes all events from the buffer.
  • Event levels: Events are color-coded by severity — info, success (green), warning (amber), and error (red).

The logger only logs meaningful state transitions. Repeated identical decisions (e.g., "holding at 0A" every 10 seconds) are deduplicated to keep the log clean.

13. Tesla Fleet API Setup

The Fleet API channel uses Tesla's cloud OAuth2 with the Vehicle Command Protocol (signed commands via tesla-http-proxy). Follow these steps if you haven't already completed them during installation.

Step 1: Register a Tesla Developer App

  1. Go to developer.tesla.com and sign in with your Tesla account.
  2. Register a new application.
  3. Set the Redirect URI to: https://my.home-assistant.io/redirect/oauth
  4. Request these scopes: openid, offline_access, vehicle_device_data, vehicle_cmds, vehicle_charging_cmds
  5. Note your Client ID and Client Secret. Add them to .env.

Step 2: Generate Fleet API Key Pair

Generate an EC private key (secp256r1/prime256v1):

openssl ecparam -name prime256v1 -genkey -noout -out tesla_fleet.key
openssl ec -in tesla_fleet.key -pubout -out tesla_fleet.pub

Host the public key at: https://your-domain/.well-known/appspecific/com.tesla.3p.public-key.pem

The private key file tesla_fleet.key must be in the project root directory. It is mounted into the HTTP proxy container to sign commands.

Step 3: Generate TLS Certs for the Proxy

mkdir -p tesla-proxy-certs
openssl ecparam -name prime256v1 -genkey -noout -out tesla-proxy-certs/tls-key.pem
openssl req -new -x509 -key tesla-proxy-certs/tls-key.pem \
  -out tesla-proxy-certs/tls-cert.pem -days 365 -subj "/CN=localhost"

Step 4: Configure .env and Start Services

TESLA_CLIENT_ID=your-client-id
TESLA_CLIENT_SECRET=your-client-secret
TESLA_PROXY_URL=https://localhost:4443
./deploy.sh

Step 5: OAuth Authorization

Go to Settings → Tesla Connection:

  1. Click Open Tesla Login — a new tab opens with Tesla's OAuth page.
  2. Sign in and approve the app permissions.
  3. After redirect, copy the code parameter from the URL.
  4. Paste it into the authorization code field and click Connect.

Tokens are stored in the SQLite database and auto-refreshed before expiry.

Key Revocation

If the proxy's IV counter gets corrupted, Tesla may revoke your key. The app detects this (2 consecutive failures on fresh sessions), auto-disables proxy commands, and logs "Key appears revoked". To recover: re-add the key via QR code in the Tesla mobile app, then click Clear Key Revoked on the Vehicle page. Consider switching to ESPHome BLE to avoid this entirely.

14. ESPHome BLE Setup

The BLE channel uses an ESP32 running ESPHome with the Tesla BLE component. Commands go directly from the ESP32 to the Tesla over Bluetooth — no Tesla cloud, no rate limits, sub-second response.

Prerequisites

  • An ESP32 board (ESP32-S3 recommended) flashed with the ESPHome Tesla BLE configuration.
  • The ESP32 must be on the same local network as your server.
  • ESPHome's native web server enabled on the ESP32 (provides the REST API the app uses).

Step 1: Flash Your ESP32

Install ESPHome and flash the Tesla BLE YAML configuration to your ESP32. The configuration must include:

  • The tesla_ble external component.
  • Sensors for: battery_level, charging_current, charging_state, range, charging_rate, energy_added, outside temperature, TPMS pressures.
  • Switches for: charging, sentry_mode, heated_steering.
  • Covers for: charge_port_door, windows, trunk, frunk.
  • Lock for: doors, charge_port_latch.
  • Climate entity: climate (provides inside temp and HVAC control).
  • Buttons for: wake_vehicle, flash_lights, sound_horn, force_data_update.
  • Number entities for: charging_amps, charge_limit.
  • The ESPHome native web server (port 80 by default).

Step 2: Pair the ESP32 with Your Tesla

Follow the ESPHome Tesla BLE component pairing instructions. The ESP32 needs to be registered as a key in your vehicle. This is a one-time process using the Tesla mobile app.

Step 3: Configure the App

On the Settings page, under Communication Channel:

  1. Select ESPHome BLE as the channel.
  2. Enter your ESP32's IP address (e.g., 192.168.1.125).
  3. If your ESPHome API requires a key, enter it in the API Key field.
  4. Click Test Connection to verify the app can reach the ESP32.
  5. Save. The app will immediately start polling via BLE.

Step 4: Activate BLE Strategy

Switch to the BLE Aggressive strategy (or create a custom one) to take advantage of the faster poll interval. Go to Settings → Settings Strategies and click Activate.

Charging amps change too frequently Stabilization delays too short Increase Ramp Up Delay and Ramp Down Delay. Also increase Battery Penalty Delay and Grid Penalty Delay to filter transient spikes. Switch to the Fleet API Default strategy if using Fleet API.
Charging starts and stops repeatedly Surplus is right at the threshold Increase Battery Protection Buffer to add more headroom. Increase High SoC Discharge Allow if battery SoC is high.
Battery draining while EV charges Algorithm reacting too slowly Reduce Ramp Up Step to slow down increases. Increase Battery Discharge Threshold to trigger penalty earlier. If on BLE, use faster poll interval.
Grid import too high during charging max_grid_import setting too generous Lower Max Grid Import (set to 0 for strict solar-only). Lower Emergency Grid Limit for a faster hard stop.
Algorithm overrides manual start/stop Mode is Solar Only, not Manual Use the Start/Stop buttons on the Dashboard — they automatically switch to Manual mode. Or click the Manual button on the Dashboard mode bar first.
Schedule doesn't activate Wrong time/day or mode not set Verify schedule is enabled. Check that days_of_week matches today (use 3-letter lowercase). The system time must be correct — check the TZ environment variable in .env.
Solar Lux not appearing on Dashboard HA integration not configured or token missing Ensure HA_TOKEN, HA_HOST, and HA_LUX_ENTITY are set in .env. After adding variables, run ./deploy.sh. Check logs for "solar_lux" to verify the fetch succeeds.
WebSocket not updating Browser connection dropped Refresh the page. The WebSocket auto-reconnects every 3 seconds.

Viewing Logs

Use the Logger tab for workflow events (algorithm decisions, Tesla commands, MQTT status). For raw container logs:

docker logs tesla-solar-charger --tail 100 -f

To check the Tesla HTTP proxy:

docker logs tesla-http-proxy --tail 50 -f

Restarting & Rebuilding

# Restart containers (preserves existing image)
docker compose restart

# Full rebuild and redeploy (after code or .env changes)
./deploy.sh
