User Manual

Tesla Solar Charger automatically manages your Tesla EV charging to maximize solar self-consumption. It reads real-time data from your Deya inverter via MQTT (Solar Assistant), calculates available surplus solar power, and dynamically adjusts charging current through the Tesla Fleet API and Vehicle Command Protocol.

Data Source

Solar Assistant publishes inverter, battery, and grid data to MQTT every ~5 seconds. The app subscribes to all solar_assistant/# topics and caches values in Redis.

Decision Engine

Every 10 seconds, the algorithm calculates surplus solar power and adjusts Tesla charging current up or down to keep grid import and battery discharge minimal.

Vehicle Commands

Commands (set amps, start/stop charging) are signed by the tesla-http-proxy using your Fleet API private key and sent via the Vehicle Command Protocol. The car is polled every 5 minutes (configurable) for SoC and state. The proxy is recreated before every command to prevent session corruption.

Hardware

Deya 3-phase inverter, 4× Pylontech US5000 batteries (20 kWh total), Tesla EV with Fleet API access.

2. Getting Started

1

Verify MQTT Connection

Open the Dashboard. If solar, battery, and grid gauges are showing live values, MQTT is connected. If everything shows 0, check that 192.168.1.50 is reachable and Solar Assistant is publishing.

2

Connect Your Tesla

Go to Settings and follow the two-step authorization flow:

  1. Click Open Tesla Login to open Tesla's sign-in page in a new tab.
  2. After signing in, you'll be redirected to a Home Assistant page. Copy the code parameter from the URL.
  3. Paste the code into the input field and click Connect.

This uses the HA OAuth redirect relay. You need both Tesla Fleet API credentials in .env and the tesla-http-proxy running. See Tesla API Setup.

3

Configure Thresholds

On the Settings page, adjust the charger parameters to match your setup: charger phases, voltage, minimum/maximum amps, and battery protection levels.

4

Plug In & Go

With the car plugged in and mode set to Solar Only, the app will automatically start charging when surplus solar is available and stop when it's not. If the car is asleep, commands will auto-wake it.

Vehicle Controls

The page provides buttons to remotely control your vehicle. All commands are sent via the signed Vehicle Command Protocol through the HTTP proxy. Available controls include:

  • Charging: Start/stop charging, set charge limit, open/close charge port, set amps
  • Climate: Start/stop HVAC, set temperatures, seat heaters, steering wheel heater, max defrost
  • Security: Lock/unlock doors, enable/disable sentry mode
  • Trunk: Open trunk, open frunk
  • Windows: Vent all windows, close all windows
  • Alerts: Flash lights, honk horn

Data refreshes automatically every 60 seconds. If the vehicle is asleep, the page shows a "sleeping" message and preserves the last known plug-in state. If the vehicle is offline, a corresponding message is shown.

6. Charging Modes

Select a mode using the buttons in the mode bar at the top of the Dashboard.

Schedule Follows configured time-based schedules. Can allow grid charging to meet a target SoC by a deadline. Falls back to solar-only behavior outside schedule windows. When you need the car charged by a specific time (e.g., "80% by 7am").

When you use the Manual Controls on the Settings page (Wake Up, Start Charge, Stop Charge, Set Amps), the system automatically switches to Manual mode to prevent the algorithm from overriding your command.

7. How the Algorithm Works

The control loop runs every 10 seconds and follows this decision process:

Step 1: Calculate Surplus

house_load = total_load - current_ev_draw
battery_available = max(0, battery_power)
solar_surplus = pv_power + battery_available - house_load - battery_protection_buffer

The current EV draw is subtracted from load to avoid counting it twice. The buffer (default 200W) prevents the battery from being drained.

Battery charging power is included as available surplus. If the battery is currently being charged at 2000W, that power could be redirected to the EV instead. This is critical for accurate surplus calculation — without it, the algorithm underestimates available solar when batteries are absorbing excess production.

Step 1b: High SoC Discharge Allowance

When the home battery SoC is above the High SoC Threshold (default 95%), a configurable Discharge Allowance (default 150W) is added to the surplus. This means a small battery discharge (up to 150W) is tolerated to keep EV charging stable, avoiding unnecessary ramp-downs from minor fluctuations when the battery is nearly full.

Step 2: Apply Penalties (with time-based filters)

Both battery and grid penalties use configurable time delays to avoid reacting to brief transient spikes:

  • Battery discharge penalty: If battery discharge exceeds the threshold (default 500W) for longer than the Battery Penalty Delay (default 30s), the excess is subtracted from available power. Once active, the penalty remains until discharge stays below the threshold for the Battery Recovery Delay (default 60s).
  • Grid import penalty: If grid import exceeds the limit (default 200W) for longer than the Grid Penalty Delay (default 30s), the excess is subtracted from available power. Once active, the penalty remains until import stays below the limit for the Grid Recovery Delay (default 60s).

This prevents short transients (e.g., a kettle boiling for 20 seconds) from causing unnecessary ramp-downs. Set delays to 0 for immediate (legacy) behavior.

Step 3: Convert to Amps

target_amps = available_watts / (voltage × phases)

For a single-phase 230V charger: 2300W surplus = 10A. For 3-phase: 2300W = ~3.3A.

Step 4: Apply Constraints & Stabilization

  • Clamped between min_charge_amps (default 6A) and max_charge_amps (default 16A)
  • If target is below minimum → stop charging (most EVs can't charge below ~6A)
  • Ramp up is gradual: max +2A per cycle (prevents oscillation)
  • Ramp down is faster: max -4A per cycle (safety priority)

Stabilization delays: After any amps change (increase or decrease), the algorithm holds that level for a configurable period before allowing another change. This prevents constant oscillation from minor solar fluctuations. Defaults: Ramp Up Delay = 120s, Ramp Down Delay = 60s. Emergency stops and stop-below-minimum bypass stabilization.

Step 5: Speculative Start

When the calculated surplus is below the minimum charging threshold but conditions suggest more solar is available, the algorithm performs a speculative start at minimum amps. This solves the "chicken and egg" problem where the inverter curtails PV production (MPPT throttling) when there isn't enough load to consume it.

Speculative start triggers when ALL of these conditions are met:

  • EV is plugged in but not currently charging
  • Home battery SoC ≥ Speculative Start SoC (default 80%)
  • PV production ≥ Speculative Min PV (default 500W)
  • Grid import is within the allowed limit

If the inverter ramps up PV to meet the new load, the next cycle will see higher surplus and continue charging. If not (e.g., it's cloudy), the algorithm will see grid import or battery drain and stop on the next cycle.

Step 6: Safety Checks

CheckConditionAction
Low battery Battery SoC < min_battery_soc (25%) Stop charging immediately. Lockout until SoC recovers to min + hysteresis (30%)
Emergency grid Grid import > emergency_grid_limit (500W) Stop charging immediately
EV not plugged in Car disconnected Hold at 0A, no commands sent

Hysteresis

When battery SoC drops below the minimum, a lockout is activated. Charging will not resume until SoC rises above min_battery_soc + battery_soc_hysteresis (default: 25% + 5% = 30%). This prevents rapid start/stop cycling.

Wake-to-Charge

If the car is asleep and there's enough solar surplus for charging, the app will wake the car to check its plug-in status. Two strategies are used:

  • Option B (primary): If the car was plugged in before going to sleep and there's solar surplus, wake immediately to start charging.
  • Option A (fallback): If the plug state is unknown (e.g., after app restart) or the car wasn't plugged in last we knew, perform a periodic wake check every 15 minutes when solar surplus is available.

The car's is_plugged_in state is preserved across sleep cycles so the algorithm knows whether waking is worthwhile.

Proxy Session Management

The tesla-http-proxy has a known IV counter bug where the session counter gets corrupted after 2-3 commands on the same session, causing IV_SMALLER_THAN_EXPECTED errors that make the car permanently revoke the key.

Prevention: The proxy container is fully recreated (stop → remove → create → start) before every command, ensuring each command gets a brand new session with IV starting at 0. An asyncio.Lock prevents concurrent recreations, and a readiness check polls the proxy's TLS port before sending the command.

The START action is split across two ticks: tick 1 sets the amps, tick 2 sends charge_start — ensuring only one proxy command per tick.

A minimum interval of 30 seconds is enforced between proxy commands to give the car time to process session teardown/setup cleanly.

API Rate Limiting

Commands are only sent to Tesla when the target amps actually changes. The vehicle is polled at the Tesla Poll Interval (default 300s / 5 minutes). With stabilization delays preventing frequent amps changes, typical daily API usage is ~300 calls, well within Tesla's Fleet API limits. The poll interval is configurable from the Settings page.

9. Home Assistant Integration

The app can optionally read sensor data from Home Assistant to improve charging decisions. Currently used to read solar lux from an Ecowitt weather station sensor.

Setup

  1. In Home Assistant, go to your Profile → Long-lived access tokens → Create Token
  2. Add these to your .env file:
    HA_URL=https://your-homeassistant-host
    HA_TOKEN=your_long_lived_access_token
  3. Restart the app. The dashboard Solar card will show lux readings when available.

How Lux Affects the Algorithm

When solar lux data is available from sensor.ecowitt_solar_lux, it modulates the algorithm's aggressiveness:

  • Low light (< Lux Stop Threshold): Stops trying solar entirely. If currently charging, sends a stop command. No speculative starts. Saves API calls during dark/overcast conditions.
  • Cloudy (< Lux Conservative Threshold): Normal surplus-based charging works, but speculative starts are disabled. The inverter is unlikely to ramp up PV in cloudy conditions.
  • Normal (between thresholds): Standard behavior, speculative starts allowed at default SoC threshold.
  • Bright sun (≥ Lux Aggressive Threshold): Speculative start SoC requirement lowered by 10%, allowing earlier/more frequent speculative starts because sustained PV production is likely.

If HA is not configured or the sensor is unavailable, the algorithm falls back to its default behavior without lux data. All lux thresholds are configurable on the Settings page.

10. Scheduling

Schedules allow you to ensure the car reaches a target SoC by a specific time, even if grid power is needed.

Creating a Schedule

On the Settings page, scroll to "Charging Schedules" and fill in:

FieldExampleDescription
NameNight chargeLabel for this schedule
Start Time22:00When this schedule becomes active
End Time06:00Deadline for reaching target SoC. Overnight schedules (start > end) are supported.
Target SoC80%Desired Tesla battery level by end time
Allow GridYesIf Yes, grid power is used when solar is insufficient to meet the deadline
Max Grid Amps16 AMaximum amps when charging from grid (only applies if Allow Grid = Yes)
Daysmon,tue,wed,thu,friComma-separated days, or * for every day. Use 3-letter lowercase: mon, tue, wed, thu, fri, sat, sun.

How Schedules Work

  1. When a schedule is active and mode is Solar Only, the system automatically switches to Schedule behavior.
  2. The app calculates the charging rate needed to reach the target SoC by the end time.
  3. If solar surplus is enough, it charges from solar (same as Solar Only mode).
  4. If solar is insufficient and "Allow Grid" is enabled, grid power is used up to the configured max grid amps.
  5. If the car already meets or exceeds the target SoC, the schedule has no effect.

Ensures at least 60% by 7am on workdays, using grid only if needed.

Solar-only charging on weekends with a higher target.

11. Logger

The Logger tab shows a real-time event log of all workflow events happening in the system. Events are stored in an in-memory ring buffer (last 500 events) and are cleared on app restart.

CategoryEvents

Features

  • Category filters: Click a category button to show only events of that type. Click "All" to show everything.
  • Auto-refresh: Enabled by default, polls for new events every 5 seconds. Toggle the checkbox to disable.
  • Clear: Removes all events from the buffer.
  • Event levels: Events are color-coded by severity — info, success (green), warning (amber), and error (red).

The logger only logs meaningful state transitions. Repeated identical decisions (e.g., "holding at 0A" every 10 seconds) are deduplicated to keep the log clean.

12. Tesla API Setup

This app uses the Tesla Fleet API (OAuth2) with the Vehicle Command Protocol (signed commands via tesla-http-proxy) to control your vehicle.

Architecture

Fleet API (Cloud)

Used for reading vehicle data (SoC, charge state, amps). Requires OAuth2 tokens.

HTTP Proxy (Local)

The tesla-http-proxy Docker container signs commands with your private key. Required for set_amps, start/stop charging.

Step 1: Register a Tesla Developer App

  1. Go to developer.tesla.com and sign in with your Tesla account.
  2. Register a new application.
  3. Set the Redirect URI to: https://my.home-assistant.io/redirect/oauth
  4. Request the following scopes: openid, offline_access, vehicle_device_data, vehicle_cmds, vehicle_charging_cmds
  5. Note your Client ID and Client Secret.

Step 2: Generate Fleet API Key Pair

Generate an EC private key (secp256r1/prime256v1):

openssl ecparam -name prime256v1 -genkey -noout -out tesla_fleet.key
openssl ec -in tesla_fleet.key -pubout -out tesla_fleet.pub

Host the public key at https://your-domain/.well-known/appspecific/com.tesla.3p.public-key.pem. The private key goes into the project root as tesla_fleet.key and is used by the HTTP proxy to sign commands.

Step 3: Generate TLS Certs for the Proxy

mkdir -p tesla-proxy-certs
openssl ecparam -name prime256v1 -genkey -noout -out tesla-proxy-certs/tls-key.pem
openssl req -new -x509 -key tesla-proxy-certs/tls-key.pem \
  -out tesla-proxy-certs/tls-cert.pem -days 365 -subj "/CN=localhost"

Step 4: Configure the App

Edit the .env file and set these values:

TESLA_CLIENT_ID=your-client-id-here
TESLA_CLIENT_SECRET=your-client-secret-here
TESLA_PROXY_URL=https://localhost:4443

Step 5: Start Services & Authorize

cd /home/vitaliy/projects/tesla-solar-charger && docker compose up -d

Then go to Settings and complete the two-step OAuth flow:

  1. Click Open Tesla Login — signs in via Tesla in a new tab.
  2. After redirect, copy the code parameter from the URL bar.
  3. Paste it into the authorization code field and click Connect.

This uses the Home Assistant OAuth redirect relay (https://my.home-assistant.io/redirect/oauth), which forwards back to your local instance.

Token Management

Tokens are stored in the SQLite database and automatically refreshed before expiry. If you ever need to re-authorize, the Connect Tesla button will appear on the Settings page when tokens are missing or expired.
